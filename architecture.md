# architecture.md

## ROY Architecture

ROY is an agent host. It is not a shell, not an IDE, not a thin chat wrapper, and not a replacement for frontier models. Its job is to take a strong model and place it inside a shaped environment where the wrong affordances do not exist, the right tools do exist, and every meaningful action passes through a policy-bearing control plane. If the concept document established the philosophy, the architecture has to make that philosophy real. The philosophical claim was that control comes from subtraction, and that the best way to control an agent is to deny the wrong fallback surfaces and replace them with better tools. The architecture therefore has to reflect a denial-first worldview all the way down. ROY cannot merely prefer structured tools. It has to be built so that structured tools are the world, and shell-shaped fallback is absent by design.

At a high level, ROY should be understood as five cooperating systems. First, there is the user-facing application: the Dioxus desktop app that presents the conversation, tool activity, permissions, artifacts, and session state. Second, there is the session engine: the orchestration layer that owns conversation history, model turns, tool requests, approvals, denials, and execution state. Third, there is the tool runtime: the registry and execution environment for all allowed actions, including filesystem tools, repo tools, issue tools, validation tools, and any custom skills you define. Fourth, there is the policy engine: the layer that decides what the agent is allowed to do, what must be denied, what requires approval, and how blocked actions are redirected. Fifth, there is the model adapter layer: the pluggable interface through which ROY talks to Claude, Codex, or other model providers. If those five layers are cleanly separated, ROY remains focused. If they bleed into one another, it becomes an ungovernable pile of chat logic.

The single most important architectural principle is that the model never receives direct shell access. This cannot be treated as a soft rule. It has to be a structural property of the system. The session engine may ask the tool runtime to execute a named tool. The tool runtime may internally call native Rust code or, when absolutely necessary, invoke trusted wrapped utilities on behalf of the host. But the model itself must never receive a raw command surface, a terminal prompt, an arbitrary subprocess runner, or a generic text channel that it can treat as a shell. If ROY leaves even a small crack here, the entire concept begins to erode. The architecture should therefore assume that shell denial is not an optional configuration toggle but part of the product’s identity.

The frontend should be a native desktop application built in Rust and Dioxus. That choice is not incidental. Dioxus gives ROY the ability to feel like a proper product instead of a developer experiment, and Rust allows the host to keep hard control over state, tool execution, permissions, and local resources. The frontend should not be designed as a fake terminal. It can borrow the emotional fluency of a terminal, especially for speed and directness, but it should present the world as a control surface. The center of the UI is the conversation pane, because natural language remains the main way the user steers the system. But surrounding that should be a persistent sense of machine state: a sidebar for sessions and workspaces, a right-hand pane or drawer for live tool calls and results, expandable artifacts for diffs and outputs, explicit permission states, and a visible ledger of what the agent tried to do versus what it was allowed to do. One of ROY’s strategic advantages should be legibility. A terminal transcript buries intention in output noise; ROY should do the opposite.

Under that UI sits the session engine, which is the real heart of the host. The session engine owns the state machine of a conversation. It knows who sent each message, which model is active, what workspace the session is bound to, which tools are currently registered, what permissions are in effect, which tool calls are in flight, what artifacts were produced, and what denials or approvals occurred during the turn. The session engine also owns the turn loop. A user message enters the system, gets added to session history, and is passed to the active model adapter along with the declared tool surface and policy context. The model responds either with ordinary assistant text or with a structured tool call request. If it asks for a tool, the session engine routes that request to the policy engine first, not directly to the tool runtime. If the action is allowed, the session engine invokes the tool runtime and then feeds the structured result back into the conversation. If the action is denied, the denial becomes a first-class event in the session and is also fed back into the model in a controlled, informative way. The agent should not merely hit a wall. It should receive a shaped response that says, in effect, “that route is unavailable here; use these other routes instead.”

This implies that ROY should model more than messages. It should model events. A session is not just a list of user and assistant chat bubbles. It is a ledger of typed events. Some events are messages. Some are tool invocation requests. Some are tool execution results. Some are denials. Some are approvals. Some are artifact creations. Some are policy changes. Some are session-level notices, such as switching models, loading a workspace, or refreshing tool availability. Representing these as first-class typed events is important because it gives ROY a much cleaner internal life. It also makes the UI far more expressive. A denied tool call can render differently from a failed one. A tool result can be collapsed into a summary or expanded into raw detail. A diff artifact can be pinned. A validation run can be associated with the tool that caused it. This event model makes the product feel intentional instead of chat-like by default.

The tool runtime should be the second major subsystem and should be implemented as a strongly typed Rust registry. A tool is not a string. It is not “something on PATH.” It is a named capability with a schema, a description, a permission profile, an execution strategy, and a structured result contract. At minimum, each tool should declare an identifier, a human-readable help description, a machine-readable input schema, a machine-readable output schema, a risk classification, and an executor implementation. The executor can be pure Rust, a wrapped external binary controlled by the host, or a bridge to another subsystem such as Ishoo. But regardless of implementation, the model sees only a clean tool interface. This keeps the agent world coherent. It no longer thinks in terms of “maybe I can hack together a command.” It thinks in terms of “these are the capabilities in this world.”

Tool categories should likely be divided into a few broad families. The first family is workspace inspection tools: list directory contents, read files, search content, inspect metadata, show diffs, inspect git status, and so on. These tools replace the discovery and inspection value that shell usually provides. The second family is workspace mutation tools: write file, patch file, rename file, move file, remove file, create directory, apply diff, and other structured edit operations. These tools replace ad hoc file mutation. The third family is validation tools: run a named check, run tests, format code, run lint, inspect failures, and fetch logs. These replace a large chunk of the “just run the command” instinct. The fourth family is domain tools: Ishoo operations, semantic map queries, repo investigation helpers, or any other custom systems that reflect the user’s real workflows. The fifth family is higher-order skill tools: compound tools that bundle multiple primitive operations into one named workflow. These are important because they are how ROY becomes not just safer than shell, but better than shell.

The architecture should distinguish clearly between a tool and a skill. A tool is a direct capability. A skill is an opinionated composition of tools with its own help surface and its own contract. For example, `read_file` is a tool, while `investigate_recent_regression` might be a skill that internally uses git history, diff inspection, semantic map lookup, and hotspot tracing. This distinction matters because it gives ROY an avenue for growth. Early versions of the host can survive on primitive tools alone, but the long-term value comes from skills that encode workflows the model would otherwise have to reinvent through repeated low-level reasoning. Skills make the agent’s world more legible and more productive. They also provide an elegant place to encode domain knowledge without changing the model itself.

The policy engine is what gives ROY its identity as a controlled host rather than just a tool-enabled chat app. The policy engine sits between the session engine and the tool runtime. Every requested action passes through it. The engine decides whether the requested tool is available in the current workspace, whether the session’s permission profile allows it, whether the action is intrinsically denied, whether it requires explicit human approval, whether it should be rate-limited, and whether the request should be transformed before execution. This transformation idea is important. A policy engine should not only say yes or no. It should be able to redirect and constrain. For example, if a model requests a tool outside the allowed workspace, the policy engine can deny the action and return a structured explanation that includes what scope is allowed. If it requests something dangerous but approval-gated, the engine can create a pending approval event instead of failing the turn outright. If it requests something that is unavailable but there is a close substitute, the denial can advertise the substitute.

Permissions themselves should be modeled explicitly and durably. ROY should not rely on ad hoc booleans scattered across the app. It should define a permission profile for each session or workspace. That profile might include capabilities such as read files, modify files, delete files, run validations, access git history, mutate issues, generate new issues, link external tools, or invoke higher-risk skills. It should also define denials that are absolute, such as arbitrary shell execution, arbitrary subprocess spawning, or unrestricted network access. Absolute denials are part of the worldview of ROY and should probably not be casually toggled inside the UI. By contrast, approval-gated permissions can be interactive. A session might allow file reads automatically, require approval for destructive file edits, and allow Ishoo issue editing but require approval for issue landing. This gives the host room to support different degrees of strictness without sacrificing its philosophical center.

A critical architectural decision is how ROY talks to models. This should be abstracted behind a provider-neutral interface. The rest of the application should not care whether the backend is Claude, Codex, or something else. It should care only that there is an adapter capable of sending a prompt context, advertising available tools, receiving ordinary assistant text, and receiving structured tool-call intents. The adapter layer should therefore define a narrow contract: start session, send turn, stream response, parse tool call, deliver tool result, and surface provider-specific metadata if necessary. That contract allows ROY to evolve independently of any one vendor. It also protects the UI and session engine from becoming polluted by provider-specific assumptions.

At the same time, the architecture should be honest about maturity. There may be multiple adapter modes over time. One mode might be a true programmatic model adapter where ROY talks to a provider API or SDK directly and receives structured tool-calling responses natively. Another mode, more experimental and less pure, might involve wrapping or proxying an existing agent client. But the architecture should strongly prefer the former because it gives ROY real control. Wrapping a stock terminal agent is tempting because it seems like a shortcut, but it risks importing the very shell-shaped assumptions ROY is trying to escape. The architecture should be written so that a provider adapter is conceptually clean even if implementation starts with some pragmatic compromises.

The storage layer should be boring and solid. ROY needs persistent sessions, saved workspaces, tool registries, policy profiles, artifacts, and UI preferences. The temptation will be to overcomplicate this with distributed state, network syncing, or overdesigned databases. That should be resisted at first. A local durable store is enough. SQLite is an obvious candidate because it is reliable, easy to ship, and fits the shape of typed events and session metadata very well. Session events can be persisted as rows with typed payloads. Artifacts can be stored as structured blobs or as references to files in an app-managed artifact directory. Workspace bindings can be keyed to canonical local paths and optional project identifiers. Tool definitions can be versioned. Policy profiles can be stored by workspace and optionally by session. None of this needs to be glamorous. It just needs to be dependable and easy to inspect during development.

Artifact handling deserves its own mention because one of ROY’s strengths should be how it treats outputs. A shell collapses everything into stdout and stderr, which forces both model and human to parse results from text streams. ROY should instead encourage tools to return typed artifacts. A diff is not “text that happens to look like a diff.” It is a diff artifact. A test run is not “just output.” It is a validation artifact with structured failures, summary counts, and raw logs attached. A generated issue draft is an issue artifact. A semantic map trace is a trace artifact. This distinction helps the UI enormously. It also reduces the amount of waste in the conversation itself, because the model does not need to dump everything back into the chat. It can create or reference artifacts that the human can inspect directly. This makes the product feel more like a workstation and less like a glorified transcript.

One subtle but important subsystem is the help and discoverability layer. If ROY wants the agent to operate well without shell, it has to make its world easy to discover. A human can hunt around awkwardly in a GUI and eventually infer what exists. A model needs a faster route. The architecture should therefore include a machine-facing capability discovery mechanism. At the start of a session, or whenever tools change, the session engine should be able to advertise the available tool surface in a compressed but structured form. The host should also provide explicit help tools such as list tools, describe tool, and perhaps list skills or show examples. If the model asks for something impossible, the denial response should not merely say no. It should say no in a way that improves future behavior. In this sense, denials are part of the teaching surface of ROY. They are not errors; they are instruction-bearing events.

Because ROY is designed to deny shell-shaped fallback, it may still need a hidden reality-based concession: a developer surface for the human operator. This should never be available to the model. But for you, the builder and power user, it may be useful to have a hidden debug console, diagnostics view, or developer-only command surface for inspecting internal state, testing tools, and diagnosing provider behavior. This is fine as long as the architecture keeps the boundary absolutely hard. There is the human’s debug world and the model’s tool world. They are not the same thing. The existence of a hidden developer console should not imply any capability leakage to the agent.

A major strength of implementing ROY in Rust is that concurrency and streaming can be handled deliberately rather than accidentally. Model responses should stream. Tool calls may be long-running. Validation tasks may produce intermediate status. Approval prompts may pause execution. The architecture should therefore treat execution as asynchronous and eventful rather than synchronous and linear. A long-running tool call should emit progress events. A streaming model response should be renderable incrementally. A pending approval should suspend only the necessary path, not freeze the whole app. Rust’s async model, combined with a disciplined event bus inside the application, can support this cleanly. Dioxus then becomes the rendering layer that reacts to durable session state instead of imperatively managing UI chaos.

If the product grows, it will likely need workspace abstraction. ROY should not treat the local filesystem as an undifferentiated space. A session should be bound to one or more declared workspaces, each with their own policy profile, tool availability, and artifact scope. This matters for safety and clarity. Many dangerous agent systems fail because the world boundary is fuzzy. In ROY, the world should be explicit. A session belongs to a workspace. Tools operate within a workspace. Policies are attached to a workspace. Artifacts are associated with a workspace. This gives both model and human a clean mental model. It also creates a natural seam for future capabilities such as per-project tool packs or per-workspace skill registries.

Architecturally, ROY should probably begin with a small, elegant core rather than a maximal one. Version one does not need to solve everything. It needs to prove the host idea. That means it needs a strong session engine, a clean tool registry, a real policy engine, one or two provider adapters, and a polished enough UI that the product already feels intentional. It does not need to become a full IDE. It does not need multi-user collaboration, cloud sync, or embedded browser automation on day one. Its success comes from taking one powerful idea seriously and implementing it end to end: the model lives in a constrained world, that world is made of tools, and the human sees the world clearly.

In concrete module terms, the Rust codebase could be organized around a handful of top-level crates or modules. The `app` layer would contain Dioxus UI code and view models. The `session` layer would contain event types, session state, turn orchestration, streaming coordination, and artifact management. The `tools` layer would contain tool schemas, registry logic, executors, and skill compositions. The `policy` layer would contain permission profiles, denial logic, approval handling, and workspace boundaries. The `providers` layer would contain Claude, Codex, and future model adapters behind a stable trait. The `storage` layer would handle SQLite persistence, artifact references, session replay, and settings. The `workspace` layer would model bound workspaces, local paths, and context manifests. The `diagnostics` layer would support logs, telemetry, developer inspection, and debugging. These are not arbitrary organizational choices. They map directly to the conceptual model of the host.

The UI layout should reinforce that architecture. The left side of the app can handle session navigation, workspaces, and perhaps skill packs or tool sets. The center should remain the conversation, because that is still the main narrative axis of a session. The right side should be the machine’s exposed inner life: tool calls, artifacts, approvals, denials, policy state, and logs. This arrangement is useful because it mirrors the host’s actual anatomy. The chat is not the whole app. It is just one surface. The right side is where control becomes visible. If the product gets this balance right, users will begin to trust it more than a terminal transcript because it explains itself as it works.

The final architectural truth is that ROY’s power will come from asymmetry. The human gets understanding, visibility, and override authority. The model gets shaped tools, bounded permissions, and no escape hatch. That asymmetry is the point. A shell pretends everyone inhabits the same machine. ROY should not pretend that. The model and the human are not peers. The human is the operator. The model is a capable actor inside a world the operator owns. If the architecture holds that line consistently, ROY will feel coherent from the inside out. If it slips and starts treating the agent like a free-roaming terminal user with a nicer skin, it will lose what makes it worth building in the first place.

That is the architecture. ROY is a desktop agent host built in Rust and Dioxus, centered on a session engine, a strongly typed tool runtime, a first-class policy engine, a provider-neutral model adapter layer, durable local storage, and a UI that makes machine behavior legible. Its defining trait is that the model is never given shell-shaped fallback. Instead, it lives in a controlled world of explicit tools, explicit permissions, explicit denials, and visible consequences. Everything else in the architecture exists to make that world powerful enough that the agent never needs to miss what was taken away.
