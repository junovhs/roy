ROY Architecture

ROY is a controlled shell host. It is not a generic terminal emulator with a nicer skin, and it is not merely a chat application with a tool registry on the side. Its defining architectural purpose is to provide a terminal-valid environment into which terminal-native agents like Claude Code and Codex can be installed, while still allowing ROY to control what exists inside that environment. The key to the whole system is that ROY must look enough like a real shell host that those agents can run inside it, but behave differently enough internally that old fallback habits break and ROY’s own tools become the actual world.

That means the center of gravity for the architecture is not “provider adapters first.” It is environment control. ROY has to own the process model, the command surface, the workspace boundary, the visible terminal interaction, and the substitution layer that decides what a command means. If it does not own those things, then it is only wrapping a shell, not replacing one.

The first architectural layer is the shell host itself. ROY needs a terminal-facing runtime capable of supporting the basic assumptions terminal agents expect: standard input and output streams, working directory semantics, environment variables, exit statuses, command lookup, and process execution flow. This does not mean ROY needs to implement every historical shell feature or compatibility quirk. It does mean it must create an environment in which an installed agent believes it is operating inside a valid terminal world. That world can be opinionated. It can be constrained. It can be custom. But it must be real enough to host software built for terminals.

The shell host should therefore be understood as a compatibility shell, not a general-purpose one. It exists to satisfy the expectations of installed agent software and route actions through ROY’s control plane. It should resolve commands through ROY-owned registries, not through blind delegation to the operating system. It should decide what exists on PATH. It should decide what command names are legal. It should decide how current working directory and workspace bindings function. Most importantly, it should decide what simply does not exist. If Bash commands, PowerShell conveniences, or arbitrary subprocess routes are not supposed to be available to the agent, ROY should not pretend to support them. It should return a clear, fast, deterministic failure that teaches the environment and points toward the intended alternatives.

The second layer is the command resolution and substitution layer. This is the actual mechanism of control. When an agent inside ROY asks to run something, ROY must decide whether that name maps to a ROY-native capability, a wrapped trusted utility, a blocked surface, or a compatibility shim. This layer is where the philosophy becomes implementation. Instead of allowing the agent to fall through to arbitrary system binaries, ROY resolves command intent against a controlled registry. Some commands may be fully native ROY tools implemented in Rust. Some may be wrappers around existing programs you trust and want to expose in a structured way. Some may be aliases that exist only to catch old habits and redirect them to better tools. And some names should be reserved solely for denial, producing helpful “not available here; use X instead” responses.

This substitution layer should be explicit and data-driven. A command definition ought to declare its name, purpose, help text, risk level, argument shape if applicable, execution backend, and visibility. Some commands may be discoverable and encouraged. Others may exist only as compatibility traps. For example, a familiar command name might deliberately return an explanation that ROY does not provide that command and offers a nearby ROY capability instead. This is not cosmetic. It is how the environment retrains the agent without needing a prompt lecture every turn.

The third layer is the tool and capability runtime. Underneath command resolution, ROY still needs real capabilities. This runtime should be implemented as a typed Rust registry where each capability has a clear contract and result shape. Some capabilities will look shell-like from the agent’s perspective because they are exposed as commands. Others may remain more structured behind the scenes and surface only through internal protocols or higher-level command bundles. What matters is that the capabilities are not arbitrary. ROY should define explicit families of behavior: workspace inspection, file mutation, validation, issue operations, repo analysis, semantic map interaction, diagnostics, and other domain-specific skills. These should return structured outputs internally even when rendered to the terminal as readable text. That gives ROY a strong bridge between terminal compatibility and product-level visibility.

A crucial design point is that ROY may expose shell-flavored commands, but those commands should terminate in ROY logic, not in a generic shell substrate. A command can feel familiar without surrendering the machine. This is how ROY can remain installable and usable for terminal-native agents without giving up its identity. It is not anti-command. It is anti-universal-fallback.

The fourth layer is the policy engine. Every resolved command or capability invocation must pass through policy before execution. Policy determines whether the command is allowed in the current workspace, whether it is intrinsically blocked, whether it requires approval, whether it should be logged at a higher level, whether its outputs should be redacted or transformed, and whether it should be rate-limited or otherwise constrained. Policy is not a sidecar. It is one of ROY’s defining organs. Shell denial belongs here, as do workspace boundaries, write permissions, destructive operations, network restrictions, and any operation class you consider too dangerous or too broad.

Policy should be attached to sessions and workspaces, not just to the app globally. A session in one workspace may be allowed to read and patch files but not perform issue-land operations. A session in another may be diagnostic-only. An embedded agent should never be allowed to infer what is legal merely by trial and error if the host already knows. ROY should tell the truth about the world through fast denials, consistent behavior, and visible policy state.

The fifth layer is workspace management. ROY’s world should not be the whole machine. It should be one or more declared workspaces with explicit roots and scope rules. Current working directory, path resolution, command visibility, and capability access should all be bound to that workspace model. This matters both for safety and for clarity. If a terminal agent inside ROY believes it has a broad machine, it will behave one way. If the environment consistently teaches that it has a scoped world, it will behave another. Workspace identity should be visible in the UI, present in session state, and respected by all command resolution and policy checks.

The sixth layer is the embedded-agent adapter layer. This is where ROY differs most from the earlier “provider adapter” picture. The primary integration target is not just a language model API. It is an installed terminal agent product such as Claude Code or Codex. ROY needs a way to install, launch, and supervise those agents inside its shell environment. That implies a dedicated subsystem responsible for starting agent processes, handling their stdin/stdout, managing authentication handoffs when needed, preserving session continuity, capturing their output as typed host events, and watching how they interact with the command world ROY provides.

This subsystem should treat embedded agents as special privileged tenants of the shell host, not as arbitrary binaries. ROY should know which installed agent is active, what version it is, how it was installed, how it authenticates, what startup contract it expects, and what session it belongs to. It should be able to start an agent process, stream its terminal interaction into the UI, detect when it attempts blocked commands, and attach those events to a durable session trace. Over time, this layer may support multiple embedded-agent profiles, but it should begin with one clean contract: launch agent, attach terminal IO, expose ROY environment, observe and record everything.

The next architectural layer is the session and transcript engine. Although ROY is shell-centered, it should not reduce everything to raw scrollback. The host still needs a strong internal model of events. A session should persist more than terminal output. It should record user inputs, agent outputs, command invocations, policy denials, approval pauses, artifacts, and host-level notices. This event model is what lets the UI become more than a terminal window. It also gives ROY replayability, debuggability, and future automation hooks. The terminal transcript is one view of the world. The event ledger is the world’s durable structure.

Artifacts deserve special treatment here. A diff should be an artifact. A validation run should be an artifact. A generated issue draft should be an artifact. A trace of denied commands should be an artifact. Even though the agent inhabits a shell, the human operator should not be forced to reason only through terminal text. ROY’s architecture should promote significant outputs into typed objects with dedicated viewers. This is one of the ways ROY becomes a workstation instead of a terminal jail.

The frontend should be built in Rust and Dioxus, but the UI should be understood as a control cockpit around the shell host. The center can absolutely include a terminal-style pane because that is the natural visual surface for an embedded terminal agent. But the UI should not stop there. It should also show session metadata, workspace context, current policy profile, a live event or activity pane, approvals and denials, artifacts, and a hidden or expandable diagnostics surface. The terminal is part of the product, not the whole product. ROY’s advantage comes from making the machine visible. The user should be able to watch the agent inhabit the shell while simultaneously seeing what the shell actually is: a controlled world.

That UI should also support explicit diagnostics because building this product without deep introspection would be miserable. A hidden developer pane should show raw command resolution, argument payloads, policy decisions, blocked-command attempts, embedded agent IO boundaries, and capability execution traces. This pane is not for the embedded agent. It is for you. Since ROY’s whole identity depends on shaping the environment correctly, you will need to see exactly how the embedded agent experiences that environment and exactly where your shell diverges from old assumptions.

Storage should remain local and boring at first. SQLite is still the right answer for sessions, events, installed-agent metadata, workspace bindings, policy profiles, and artifact references. The important difference now is that storage must also understand embedded-agent sessions and shell traces, not just tool-driven chat sessions. A reopened ROY session should be able to show what the agent tried, what commands resolved, what was denied, and what outputs were produced. That history is not just useful for debugging. It is part of the product’s trust model.

A subtle but central requirement is discoverability from inside the shell. Since ROY is intentionally denying old habits, it must make the new world easy to learn. This means providing clear help surfaces through commands the embedded agent can reach naturally. There should be obvious built-ins for listing available commands, describing ROY-native capabilities, understanding workspace scope, and learning which denied habits have approved replacements. If an embedded agent tries grep and gets denied, the denial should be immediate and informative. It should not simply fail. It should say, in effect, that this environment does not provide that surface and that these other commands exist instead. Over time, this discoverability layer becomes one of the ways the shell retrains the agent.

From an implementation standpoint, the Rust codebase should reflect this architecture directly. There should be a shell module for the terminal host, process environment, command resolution, PATH shaping, and IO handling. There should be a commands or capabilities module for the actual ROY-native commands and wrapped utilities. There should be a policy module for permissions, denials, approvals, and scope enforcement. There should be a workspace module for roots, current directory semantics, and path normalization. There should be an agents module for installed terminal-agent adapters like Claude Code and Codex. There should be a session module for typed events and durable history. There should be a storage module for SQLite persistence. There should be a ui module for the Dioxus frontend and its diagnostic surfaces. These are not arbitrary names. They are the skeleton of the product.

The v0.1 target should be reinterpreted through this architecture. A convincing v0.1 is not “ROY can talk to a model API.” It is “ROY can host at least one real terminal-native agent inside its shell, deny shell-shaped fallback, expose ROY-native commands instead, and let a user complete a simple task while watching the interaction through ROY’s UI.” That is the thesis made real. API integrations can still exist later if they are useful, but they are no longer the center of the architecture. The shell host is the center.

That is the architectural picture. ROY is a custom shell environment with a policy-bearing command world, typed capabilities underneath that world, explicit workspace boundaries, embedded-agent adapters for tools like Claude Code and Codex, a durable session and artifact ledger, and a Dioxus desktop UI that reveals the machine rather than hiding it. It wins not by becoming a better Bash, but by becoming a valid shell whose universe is finally under your control.
